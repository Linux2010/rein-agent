## 二、架构模式选择

### 2.1 三种主流架构

#### 模式 A：网关路由式（OpenClaw 模式）

```
用户 → 渠道层 → 网关(Gateway) → Agent 路由 → 多 Agent → 工具调用
                    ↓
              记忆/会话/配置
```

**优势**：
- 渠道无关，支持 Telegram/Discord/Web 等
- Agent 隔离，独立运行
- 插件化扩展
- 本地部署，隐私保护

**劣势**：
- 架构复杂，维护成本高
- 需要处理并发和路由

#### 模式 B：IDE 集成式（Claude Code 模式）

```
用户 ↔ IDE/编辑器 ↔ AI Agent ↔ 文件系统 ↔ 终端
```

**优势**：
- 深度 IDE 集成，开发体验好
- 上下文感知强
- 实时反馈

**劣势**：
- 绑定 IDE 生态
- 不具备多渠道能力
- 依赖商业 API

#### 模式 C：自主执行式（AutoGPT 模式）

```
目标 → Agent → 规划 → 工具调用 → 验证 → 循环
```

**优势**：
- 高度自主
- 适合复杂任务链
- 框架灵活

**劣势**：
- 容易失控（无限循环）
- 成本不可控
- 缺乏人类监督机制

### 2.2 推荐方案：混合架构

```
┌─────────────────────────────────────────┐
│               交互层 (Channels)          │
│  Telegram │ Discord │ Web │ CLI │ Voice │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│           网关层 (Gateway)               │
│  认证 │ 路由 │ 会话 │ 心跳 │ 定时任务    │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│          Agent 编排层 (Brain)            │
│  Leader → Worker → 任务分发 → 状态管理   │
└─────────┬─────────┬─────────┬──────────┘
          │         │         │
┌─────────▼┐ ┌──────▼──┐ ┌────▼──────┐
│ Coder    │ │Analyst  │ │Ops Agent  │
│ 编码专家 │ │分析专家 │ │运维专家   │
└────┬─────┘ └────┬────┘ └────┬──────┘
     │            │            │
┌────▼────────────▼────────────▼──────┐
│          工具层 (Skills + Tools)      │
│  Web搜索 │ 浏览器 │ Git │ 数据库 │ MCP│
└─────────────────┬───────────────────┘
                  │
┌─────────────────▼───────────────────┐
│          记忆层 (Memory)              │
│  短期记忆 │ 长期记忆 │ 向量检索 │ 档案│
└─────────────────────────────────────┘
```

## 三、技术栈选型

### 3.1 运行时

| 选项 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Node.js** | 生态成熟、TS 原生支持、异步强 | 单线程、CPU 密集弱 | ⭐⭐⭐⭐⭐ |
| **Python** | AI 生态强、库丰富 | 异步弱、类型系统差 | ⭐⭐⭐⭐ |
| **Go** | 并发强、性能好 | 泛型弱、生态少 | ⭐⭐⭐ |
| **Rust** | 性能极致、安全 | 学习曲线陡、开发慢 | ⭐⭐ |

**结论：Node.js + TypeScript**
- 与 OpenClaw 生态一致
- TypeScript 类型安全
- 异步 IO 适合 Agent 场景
- npm 生态丰富

### 3.2 AI 模型集成

| 模型 | 优势 | 劣势 | 使用场景 |
|------|------|------|---------|
| **Qwen 系列** | 中文强、开源、成本低 | 推理弱于 GPT | 日常对话、任务执行 |
| **Claude** | 推理强、代码好 | 贵、API 限制 | 复杂编码任务 |
| **GPT-4o** | 通用最强 | 贵、延迟高 | 关键决策 |
| **GLM 系列** | 中文好、免费额度 | 生态弱 | 轻量任务 |
| **本地模型** | 隐私、免费 | 性能有限 | 离线场景 |

**结论：多模型支持 + 智能路由**
```
简单任务 → Qwen/GLM（低成本）
复杂编码 → Claude（高质量）
关键决策 → GPT-4o（高可靠）
离线场景 → 本地模型（隐私）
```

### 3.3 核心依赖

| 组件 | 推荐方案 | 替代方案 |
|------|---------|---------|
| **LLM SDK** | OpenAI SDK（统一接口） | LangChain |
| **向量存储** | sqlite-vec（嵌入式） | LanceDB、ChromaDB |
| **任务队列** | BullMQ（Redis） | 内置 Cron |
| **HTTP 客户端** | undici / axios | node-fetch |
| **CLI 框架** | commander | oclif |
| **配置管理** | Zod + YAML | json5 |
| **日志** | tslog / pino | winston |
| **测试** | Vitest | Jest |
| **构建** | tsup / tsdown | tsc |

## 四、关键设计决策

### 4.1 Agent 粒度

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **单 Agent 多技能** | 一个大 Agent，通过技能切换 | 个人助手 |
| **多 Agent 协作** | 多个专用 Agent，分工合作 | 团队场景 |
| **混合模式** | Leader 协调 + Worker 执行 | **推荐** |

### 4.2 记忆架构

```
┌─────────────────────────────┐
│      工作记忆 (Working)      │ ← 当前任务上下文
├─────────────────────────────┤
│      短期记忆 (Short-term)   │ ← 最近 24h 对话
├─────────────────────────────┤
│      长期记忆 (Long-term)    │ ← 向量索引 + 文本档案
├─────────────────────────────┤
│      技能记忆 (Skill Memory)  │ ← 学习到的模式和经验
└─────────────────────────────┘
```

### 4.3 安全边界

| 层级 | 保护措施 |
|------|---------|
| **输入** | 内容过滤、注入防护 |
| **执行** | 沙箱隔离、权限控制 |
| **输出** | 内容审核、格式校验 |
| **存储** | 加密、访问控制 |
| **网络** | 白名单、速率限制 |

## 六、与 OpenClaw 的差异化

| 维度 | OpenClaw | x-agent |
|------|---------|---------|
| **定位** | 个人 AI 助手平台 | 通用 Agent 智能体框架 |
| **部署** | 本地优先 | 本地 + 云端 |
| **Agent** | 多实例独立运行 | 框架内多 Agent 协作 |
| **扩展** | 技能 + 插件 | 技能 + Agent 类型 |
| **目标** | 个人日常助手 | 企业/个人通用智能体 |
| **代码能力** | 基础 | 深度集成 Claude Code |
| **任务系统** | OpenTask 集成 | 原生任务引擎 |

**核心差异**：x-agent 不重复造轮子，而是作为 OpenClaw 的补充，提供更轻量、更专注的 Agent 框架。

## 八、结论与建议

### 推荐技术栈

```
运行时：Node.js 24 + TypeScript 5
LLM：多模型路由（Qwen/GLM/Claude）
存储：sqlite-vec（嵌入式）
构建：tsdown + pnpm
测试：Vitest
部署：Docker + GitHub Actions
```

### 关键原则

1. **轻量优先** - 不重复造轮子，复用 OpenClaw 生态
2. **插件化** - 一切皆插件，保持核心精简
3. **本地优先** - 默认本地运行，可选云端
4. **安全边界** - 沙箱隔离，权限控制
5. **渐进增强** - 从简单开始，按需扩展
